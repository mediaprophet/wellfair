# Critical Review & Fix List – wellfair Project

**Date:** 28 May 2026  
**Author:** Timothy Holborn  
**Purpose:** This document identifies the major mismatches, inconsistencies, and gaps between the **current code/documentation** and the **intended broader vision** of wellfair. It will be used to direct an agent (or developer) to systematically update the project.

### Executive Summary

The project has **significantly evolved** beyond its original “health-to-solid” scope (Samsung Health exports → Solid-compatible RDF). It is now intended to become **wellfair** — a Personal Wellbeing Informatics Vault focused on information critical to life and welfare on fair terms, with health-to-solid remaining a core but expandable component.

However, the **public narrative, documentation, and presentation** have not kept up with this evolution. The repository currently sends mixed and outdated messages:

- GitHub repo description is broad and philosophical.
- README.md and most documentation still present the project as a narrow Samsung Health converter called **health-to-solid**.
- Advanced features, welfare modelling, and concepts such as **Sanctuary Mode** are either missing or invisible to outsiders.

This creates confusion for new visitors, potential collaborators, and future users.

### Major Issues Identified

#### 1. Narrative & Project Identity (Highest Priority)
- The project is still branded and described everywhere as **“health-to-solid”**.
- The name **“wellfair”** does not appear in the README.md at all.
- There is no clear explanation of the evolution: started as health-to-solid → has grown into the broader wellfair vision.
- No mention that health-to-solid remains fully supported and will be expanded (especially into welfare-related factors evaluated against models such as Maslow’s hierarchy of needs).
- No statement that the project has **no connection** to any previous HealthPod work.

#### 2. Documentation Gaps
- README.md is heavily focused on Samsung Health pipeline and technical details.
- Broader objectives (wellbeing informatics, fair-terms, welfare modelling, Sanctuary Mode, etc.) are absent or only lightly implied.
- Future extensions are listed but feel like add-ons rather than part of a coherent larger mission.
- No high-level “About Wellfair” or “Project Vision” section that clearly communicates the expanded scope.

#### 3. UI / Application Issues
- Settings panel in the Streamlit app (`ui/app.py`) is still blank / unimplemented, despite references in `NAVIGATION_UPDATE.md` to a “Dedicated Settings Menu”.
- The app interface does not yet reflect the broader wellfair identity.

#### 4. Deployment & Demo
- GitHub Pages (WASM demo in `web/`) is almost ready but not activated (requires changing Pages source to “GitHub Actions” in repo Settings).
- The live demo (once deployed) will only show the original health-to-solid converter, not the expanded wellfair scope.

#### 5. Architectural & Philosophical Clarity
- Growing functionality (including Sanctuary Mode and deeper welfare analytics) raises legitimate questions about full Solid compatibility.
- There is currently no documentation distinguishing between:
  - What is Solid-native / Solid-compatible.
  - What is wellfair-specific / local-first / extended beyond standard Solid.
- No explicit guidance yet on how welfare factors (e.g. Maslow’s hierarchy) will be modelled and evaluated within the ontology.

#### 6. Code & Folder Structure
- Many internal references, folder names, and comments still revolve heavily around the original “health-to-solid” concept.
- The shift to `wellfare-core` is a start, but the broader rebranding is incomplete.

### Recommended Fix Priority

1. **Repositioning & Narrative** – Update README, repo description, and key docs to reflect the true wellfair vision while keeping health-to-solid as a supported core capability.
2. **Implement Settings Panel** – Make the dedicated settings menu functional.
3. **Activate GitHub Pages** – Get the WASM demo live and update its messaging.
4. **Clarify Solid Boundaries** – Document what is Solid-compatible vs wellfair-extended (including Sanctuary Mode).
5. **Ontology & Welfare Expansion** – Begin documenting how welfare models (Maslow etc.) will be integrated.
6. **Consistent Branding** – Ensure “wellfair” appears prominently and correctly throughout the project.
